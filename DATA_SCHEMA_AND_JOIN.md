## Tracking JSON Schema

Top level:

```json
{
  "gameid": "0021500504",
  "gamedate": "2016-01-02",
  "events": [ ... ]
}
```

Each `event`:

- `eventId` (string): event number for this game (join key to PBP `EVENTNUM`)
- `visitor` (object): team + roster info
- `home` (object): team + roster info
- `moments` (list): high-frequency tracking frames

Each `moment` is a fixed-order array:

- `moment[0]`: `quarter` (int)
- `moment[1]`: wall-clock timestamp in ms (int)
- `moment[2]`: `game_clock` in seconds (float)
- `moment[3]`: `shot_clock` in seconds (float or null)
- `moment[4]`: unused/null in this dataset
- `moment[5]`: list of tracked entities

Tracked entity row format:

- `[team_id, player_id, x, y, z]`

Ball row conventions:

- Ball is encoded as `[-1, -1, x, y, z]`
- When present in a normal frame, it is at index `moment[5][0]`

## PBP CSV Schema (Important Columns)

From `2015-16_pbp.csv`, the most useful join/metadata columns are:

- `GAME_ID`: game id (matches JSON `gameid`)
- `EVENTNUM`: event number (matches JSON `eventId`)
- `EVENTMSGTYPE`: event type code (made shot, miss, rebound, foul, etc.)
- `EVENTMSGACTIONTYPE`: subtype code
- `PERIOD`, `PCTIMESTRING`: period and game clock string
- `HOMEDESCRIPTION`, `VISITORDESCRIPTION`, `NEUTRALDESCRIPTION`: text descriptions
- `SCORE`, `SCOREMARGIN`: scoreboard state
- `PLAYER1_*`, `PLAYER2_*`, `PLAYER3_*`: actor/team metadata

Note: the CSV includes an unnamed leading index column (`""`) that should be dropped.

## Join Logic

Join keys:

- `tracking.gameid == pbp.GAME_ID`
- `int(event.eventId) == int(pbp.EVENTNUM)`

Practical algorithm:

1. Filter PBP rows to one `GAME_ID`.
2. Build map: `EVENTNUM -> pbp_row`.
3. For each tracking event, look up by `eventId`.
4. Attach `pbp` row (or `null` if not found).

## Moment-Level Edge Cases

Do not assume every moment has exactly 11 rows.

In `0021500504.json`:

- `11` rows (ball + 10 players): `240157` moments
- `10` rows (usually players only, no ball row): `1697` moments
- `1` row (ball only): `7` moments

Recommendation:

- Always parse entity rows dynamically.
- Detect ball row by `team_id == -1 and player_id == -1` instead of hardcoding index `0`.

## Event, Moment, and Outcome Relationships

This section explains how to connect tracking frames to basketball outcomes.

### Relationship Hierarchy

- `game` contains many `events`
- each `event` contains many `moments`
- each `moment` is a frame-level snapshot of player/ball positions

So the data is hierarchical:

- game-level outcomes (win/loss, final score)
- possession-level outcomes (made shot, turnover, foul, free throws, etc.)
- frame-level states (spacing, velocity proxies, shot clock context)

### Key Point: Outcomes Are Event-Level Labels, Not Moment-Level Labels

- `moments` do not directly say "this frame is a make/turnover"
- outcome labels come from joined PBP fields (`EVENTMSGTYPE`, descriptions, score changes)
- after joining, the event is the natural unit for supervised labels

### Practical Correlation Strategy

1. Build event-level labels from PBP.
2. Aggregate moment-level features within each event.
3. Correlate event features with event outcomes.

Example features:

- event duration
- average ball height
- spacing / nearest-defender distance
- shot-clock context

### Possession Outcomes

Events are not always equal to possessions.

- A possession can include multiple events (pass, shot, rebound, foul sequence)
- A single event can be one action inside a larger possession

Recommended possession workflow:

1. Order joined events by period + clock/event order.
2. Build possession segments using PBP rules (made basket, defensive rebound, live-ball turnover, etc.).
3. Assign each event/moment to a possession id.
4. Compute possession-level features from all moments in that possession.
5. Attach possession outcome label (0/1/2/3 points, turnover, foul drawn, etc.).

### Game Outcomes

To relate tracking to game outcomes:

- aggregate possession/event metrics to team-game summaries
- compare distributions for winning vs losing teams
- include game-state context from PBP score/margin (clutch, garbage time, foul game)

### Modeling Guidance

- Use `eventId <-> EVENTNUM` join first; treat PBP as ground-truth labels.
- Keep unmatched events (`[194, 207, 404]` in this game) and handle explicitly.
- Parse frames robustly (ball can be missing in some moments).
- Use multi-level pipelines:
  - moment -> event features
  - event -> possession features
  - possession -> game outcome features
